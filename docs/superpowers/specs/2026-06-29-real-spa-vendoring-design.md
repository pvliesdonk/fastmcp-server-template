# Design: real SPA SDK vendoring in the template

- **Date:** 2026-06-29
- **Status:** approved (brainstorming) ‚Äî pending implementation plan
- **Related issues:** #216 (contradictory SPA-vendoring guidance + dangling
  `app.src.html` reference); the SPA-scaffold leak
  issue #218; CI `--check` wiring #219; deferred downstream-convergence
  follow-ups markdown-vault #765 and image-generation #292.

## Problem

The template ships an SPA-vendoring scaffold that is a **no-op placeholder**:
`scripts/vendor_spa.py` does nothing, `static/app.html` is an unwired hand-authored
shell, and `.gitattributes` declares `app.html` "Generated by scripts/vendor_spa.py
‚Äî do not edit directly. Edit static/app.src.html instead" while no `app.src.html`
is ever generated and nothing loads `app.html` (the working demo serves HTML inline
from `_SPA_SHELL_HTML` in `_server_apps.py`). The three artifacts describe two
incompatible models and reference a file the template never ships (#216).

This is not merely a documentation defect. `fastmcp-server-template` is the
**shared opinionated core for a fleet of MCP servers**, not a kickstart. A vague
placeholder is under-specification, and under-specification produces downstream
divergence debt: the two current apps-capable consumers independently invented
**two different solutions to the same problem**.

### Fleet state at design time (verified)

| | markdown-vault | image-generation | template (current) |
|---|---|---|---|
| Vendors | `@modelcontextprotocol/ext-apps@1.3.1` (+ domain libs) | `ext-apps@1.3.1` (same sha256) | nothing (no-op) |
| Packaging | inlined into `app.html` (~1.19 MB blob) | base64 into `_vendored_sdk.py` (~430 KB) | inline `_SPA_SHELL_HTML` string |
| Source | `app.src.html` (~52 KB, CDN `<script src>` tags) | `vendor_sdk.py` config | ‚Äî |
| Serves via | `importlib.resources` (static file) | Python module import | inline string |
| Drift check | `--check` + `<!-- vendor-spa-source-sha256:‚Ä¶ -->` marker | `config-hash` + `content-sha256` markers | none |

Both downstreams download the **same** `@modelcontextprotocol/ext-apps` SDK, pin
it, SHA256-verify it, and embed a drift-detection marker. The universal need is
identical ‚Äî *vendor `ext-apps`, pinned + hash-verified, with a CI drift check*.
They disagree only on **packaging** (inline-in-HTML vs. base64-Python-module).

## Decisions

1. **Packaging model: HTML-inline (markdown-vault's model).** `vendor_spa.py`
   inlines the SDK into `static/app.html`; `_server_apps.py` serves that static
   file via `importlib.resources`. Two-file `app.src.html` ‚Üí `app.html`. Chosen
   over the Python-module model because it is the more mature, battle-tested
   implementation, the SPA HTML is the natural home for a frontend SDK, and
   static serving via `importlib.resources` is the conventional pattern.

2. **Scope: template only; downstream convergences tracked and deferred.** Land
   the canonical implementation in the template first; converge the fleet
   afterward via copier update.

3. **Bootstrap: generate `app.html` at scaffold via copier `_tasks`.** The
   template ships no committed SDK blob; a post-generation task runs
   `vendor_spa.py` after render to produce `app.html` in the new project. Cost
   accepted: network access (download `ext-apps`) at `copier copy`/`update` time.

4. **Generic static HTML + fixed source token.** The SPA HTML carries no
   per-project Jinja. It uses a fixed source token `app___<tool>`; all
   per-project specificity (module name, app name, human name, tool
   registrations) lives in the templated `_server_apps.py`. This keeps Jinja
   away from the minified SDK blob entirely (minified JS contains `{{`/`{%`
   sequences that would break templating).

## Architecture

### Ownership and gating

| File | Templated | Gated on `include_mcp_apps_scaffold` | Ownership |
|---|---|---|---|
| `scripts/vendor_spa.py` | no (static `.py`) | yes (currently ungated ‚Äî leak fix) | template-owned + `DOMAIN-VENDOR-LIBS` block |
| `static/app.src.html` | no (static, generic) | yes | domain-owned ‚Üí `_skip_if_exists` |
| `static/app.html` | not shipped | n/a | project-side, generated by `_task`, committed downstream |
| `.gitattributes` | yes | yes (currently ungated ‚Äî leak fix) | template-owned |
| `_server_apps.py` | yes | yes (already) | template-owned + `DOMAIN-APP-*` blocks |
| `copier.yml` `_tasks` | yes (Jinja-gated) | yes | template-owned |

The current `.gitattributes` is 100% SPA lines, so gating the whole file is
correct: apps-OFF projects receive none of it (today they wrongly receive
`app.html`, `.gitattributes`, and `vendor_spa.py` ‚Äî the leak).

**Gating mechanism for non-`.jinja` files.** `_server_apps.py` is gated today
via an in-file `{% if %}` (it renders inert content when off), but that does not
remove a file from the output ‚Äî wrong for `vendor_spa.py`, `app.src.html`, and
`.gitattributes`, which must be *absent* in apps-OFF projects. copier removes a
file whose rendered path is empty, so these are gated by a **conditional path
segment** in the filename, e.g.
`{% if include_mcp_apps_scaffold %}vendor_spa.py{% endif %}` (path templated,
content copied verbatim ‚Äî no `.jinja` content-suffix, so copier does not try to
template the file body), and likewise for `app.src.html` and `.gitattributes`.
The exact spelling (conditional filename vs. a Jinja `_exclude` expression) is an
implementation-plan choice; the requirement is that apps-OFF output contains none
of these files.

### `vendor_spa.py` contract

Generalized from markdown-vault's implementation; **stdlib-only** (`urllib`,
`hashlib`, `base64`, `re`, `pathlib`).

- `VENDORED_VERSIONS` ships **only** `@modelcontextprotocol/ext-apps` (pinned
  version + unpkg URL + sha256 + `type: module` + import specifier), wrapped in a
  `# DOMAIN-VENDOR-LIBS-START` / `# DOMAIN-VENDOR-LIBS-END` marker block
  preserved across copier update, where a project adds its own libs.
- Default invocation (no args) regenerates `static/app.html`: read
  `static/app.src.html`, download each pinned lib, **sha256-verify** against the
  manifest (hard-fail on mismatch), inline the `<script src>` / module tags as
  inline `<script>`, write `static/app.html`, embed a
  `<!-- vendor-spa-source-sha256:<hash> -->` marker.
- `--check`: recompute the source hash from `app.src.html`, compare to the marker
  embedded in `app.html`; non-zero exit on drift. **Offline** ‚Äî no network, pure
  local comparison.

### `_server_apps.py` ON-branch change (inline ‚Üí static)

- Remove the inline `_SPA_SHELL_HTML = """‚Ä¶"""`. Load instead via
  `importlib.resources.files(<package>).joinpath("static/app.html").read_text()`.
- Keep `_rewrite_spa_app_tool_calls` and the three-way `_APP_TOOL_NAMES`
  validation, now run against the **loaded `app.html`** content. The rewriter
  matches the fixed `app___<tool>` source token and maps it to
  `hashed_backend_name(<python_module>, tool)`; the wire app-name remains
  `python_module`.
- The `DOMAIN-APP-HTML` block moves out of `_server_apps.py` into
  `static/app.src.html` (domain-owned). `DOMAIN-APP-TOOL-NAMES` and
  `DOMAIN-APP-TOOLS` stay in `_server_apps.py`.

### `static/app.src.html` starter

A generic demo SPA equivalent to the current inline shell: a minimal page with
buttons that call the placeholder app-only tools (`get_status`, `get_info`) via
`app.callServerTool`, using fixed `app___get_status` / `app___get_info` source
literals and a `<script type="module" src="‚Ä¶ext-apps‚Ä¶">` tag that `vendor_spa.py`
inlines. Generic title; no Jinja.

### Bootstrap via `copier.yml` `_tasks`

```yaml
_tasks:
  - "{% if include_mcp_apps_scaffold %}python scripts/vendor_spa.py{% endif %}"
```

Task values are Jinja-rendered; an entry rendering to empty string is skipped, so
the task only runs for apps-ON projects. Runs on both `copier copy` and
`copier update` (`_min_copier_version: 9.0`; both CI paths already pass
`--trust`). On update it regenerates `app.html` from the project's
possibly-customized `app.src.html`. Offline ‚Üí task fails (accepted trade-off).

## Testing

- **Generated project CI gate:** add `python scripts/vendor_spa.py --check`
  (offline, gated on apps) as the anti-drift enforcement; optionally a matching
  pre-commit hook.
- **`template-ci.yml` self-test:** smoke-answers has `include_mcp_apps_scaffold:
  true`, so the `_task` runs at render (needs network), producing `app.html`;
  the generated gate's `--check` must then pass.
- **Rendered-project test:** assert `static/app.html` exists and is
  `--check`-clean; assert `_server_apps.py` loads it via `importlib.resources`
  and the `_rewrite_spa_app_tool_calls` three-way validation still holds against
  the loaded content.

## Implementation breakdown (epic)

Active (template):

1. **Real two-file vendoring pipeline** ‚Äî `vendor_spa.py` (real, ext-apps
   pinned+sha256, `--check`), generic static `app.src.html`, `_server_apps.py`
   switched to `importlib.resources` static serving with the fixed `app___<tool>`
   token, `copier.yml` `_task` to generate `app.html` at scaffold.
   **Closes #216.**
2. **Gate the SPA scaffold + fix the leak** ‚Äî gate `vendor_spa.py`,
   `app.src.html`, `.gitattributes`, and the `_task` behind
   `include_mcp_apps_scaffold`; `app.src.html` ‚Üí `_skip_if_exists`. Apps-OFF
   projects receive none of it. **Closes #218** (the SPA-scaffold leak).
3. **CI `--check` wiring** (#219) ‚Äî generated project's CI gate runs
   `vendor_spa.py --check`; template-ci self-test asserts `app.html` exists and
   is check-clean.

Ordering 1 ‚Üí 2 ‚Üí 3 (issue 2 re-shapes files issue 1 creates; 1+2 may merge at
plan time if cleaner).

Deferred (filed, tracked):

4. **markdown-vault convergence** (#765, light) ‚Äî adopt the template-owned
   `vendor_spa.py`, keep domain libs (vis-network/marked/dompurify) in the
   `DOMAIN-VENDOR-LIBS` block.
5. **image-generation migration** (#292, heavy) ‚Äî switch `_server_apps.py` to
   HTML-inline static serving, adopt the two-file scaffold, **delete**
   `_vendored_sdk.py` and `vendor_sdk.py` (removal-discipline verification).

## Risks and notes

- **Network at scaffold/update.** The `_task` and the downstream weekly
  copier-update CI now require network to download `ext-apps`. Offline copy/update
  fails the task. Accepted per decision 3.
- **template-ci network dependency.** The self-test render now downloads
  `ext-apps`. If unpkg is flaky, template-ci flakes; mitigation options (retry,
  cache) are an implementation-plan concern, not a design change.
- **SDK pin maintenance.** `ext-apps` version + sha256 live in the template's
  `vendor_spa.py`; bumping is a template-side change that propagates via copier
  update ‚Äî which is the point (single-point fleet maintenance).
- **`copier update` regenerates `app.html` every run.** Idempotent given a pinned
  version and unchanged `app.src.html`; a customized `app.src.html`
  (`_skip_if_exists`, domain-owned) drives a correct regeneration.
